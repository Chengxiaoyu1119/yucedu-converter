# 项目结构、技术栈与路径规范

本文帮助第一次接触项目的人快速理解：每个文件放在哪里、使用了什么技术、一次转换怎样完成，以及后续开发应遵守哪些路径规则。

## 一、整体结构

仓库只保存可以公开、可以复现的源码、文档、测试和构建配置。本机样本、参考播放器、转换结果与生成文件不进入 Git。

```text
yucedu-converter/
├─ .github/                         # GitHub 协作、持续集成与项目规范
│  ├─ ISSUE_TEMPLATE/               # 问题与功能建议表单
│  ├─ workflows/                    # Windows 自动检查与发布流程
│  ├─ CONTRIBUTING.md               # 开发和提交规范
│  ├─ SECURITY.md                   # 安全问题报告方式
│  └─ pull_request_template.md      # Pull Request 检查清单
├─ docs/                            # 用户文档、维护文档与研究资料
│  ├─ images/                       # README 和说明文档使用的图片
│  ├─ 研究/                         # 上游来源与兼容性研究记录
│  ├─ 项目结构.md                   # 当前文件：结构、技术栈与路径规则
│  ├─ 更新日志.md                   # 各版本变化
│  └─ 第三方组件.md                 # 依赖及其许可证说明
├─ packaging/
│  └─ windows/                      # PyInstaller 的 Windows 打包配置
├─ scripts/                         # 构建、发布和项目检查脚本
├─ src/
│  └─ yucedu_converter/             # 正式 Python 程序包
│     └─ resources/                 # 转换资源、兼容数据和程序图标
├─ tests/                           # unittest 自动测试与合成夹具说明
├─ local/                           # 本机私有资料，已被 Git 忽略
├─ .editorconfig                    # 编辑器通用缩进与换行规则
├─ .gitattributes                   # Git 文本换行与二进制文件规则
├─ .gitignore                       # Git 排除规则
├─ LICENSE                          # MIT 许可证
├─ README.md                        # GitHub 项目首页和快速入口
└─ pyproject.toml                   # Python 项目、依赖、版本与打包元数据
```

根目录只保留项目入口和工具必须识别的配置文件。说明类文件集中到 `docs/`，GitHub 社区文件集中到 `.github/`。

## 二、技术栈

“技术栈”就是项目使用的一组语言、界面库、依赖和构建工具。

| 技术 | 在项目中的作用 | 对使用者的影响 |
|---|---|---|
| Python 3.11+ | 编写转换核心、界面和播放器调用逻辑 | 源码开发需要安装 Python；正式版已打包运行环境 |
| Tkinter / ttk | 构建 Windows 桌面图形界面 | Python 自带，不引入网页前端或浏览器组件 |
| `cryptography` | 提供 AES 等密码学运算 | 是运行时唯一的外部 Python 依赖 |
| `threading` + `queue` | 在后台处理转换，并把进度传回界面 | 大文件处理时界面仍可操作 |
| PyInstaller | 把 Python 程序打包为 Windows 可执行目录 | 正式版用户不需要单独配置 Python |
| PowerShell | 执行 Windows 构建与发布脚本 | 开发者可用一条脚本命令生成交付包 |
| `unittest` | 执行程序自带的自动测试 | Python 标准库自带，不增加测试依赖 |
| GitHub Actions | 在 GitHub 上执行 Windows 检查与发布流程 | 提交和发版过程更容易复现 |

项目当前没有数据库、服务器后端、Electron 或网页前端；主要形态是本地 Windows 桌面程序。

## 三、源码文件作用

| 路径 | 作用 |
|---|---|
| `src/yucedu_converter/__init__.py` | 保存程序名称、版本号和当前已验证配置标识 |
| `src/yucedu_converter/__main__.py` | 支持 `python -m yucedu_converter` 启动方式 |
| `src/yucedu_converter/main.py` | 程序入口；设置 Windows DPI 与任务栏应用标识后启动界面 |
| `src/yucedu_converter/gui.py` | 主窗口、任务列表、批量处理、进度显示和用户交互 |
| `src/yucedu_converter/converter.py` | 文件识别、输入检查、流式加解密、输出命名与 SHA256 校验 |
| `src/yucedu_converter/player.py` | 查找并调用 PotPlayer、VLC、Windows 默认播放器或原播放器 |
| `src/yucedu_converter/settings.py` | 设置数据结构、读取保存、资源路径和日志路径 |
| `src/yucedu_converter/theme.py` | 颜色、字体、按钮、进度条和滚动条等视觉样式 |
| `src/yucedu_converter/resources/aes_tail_table.bin` | 当前已验证转换配置使用的尾部查表资源 |
| `src/yucedu_converter/resources/compatibility_trailer.bin` | 兼容输出使用的固定尾部资源 |
| `src/yucedu_converter/resources/app.ico` | 窗口、任务栏和打包程序图标 |

当前维护策略是不拆分 `gui.py` 与 `converter.py`；路径规范化只整理文件位置和文档，不改变程序行为。

## 四、辅助目录作用

### `.github/`

只放 GitHub 能直接识别的协作文件：Issue 表单、Pull Request 模板、工作流、贡献规范和安全说明。把这些文件放在这里可以减少根目录文件，同时保持 GitHub 自动识别。

### `docs/`

面向用户和开发者的说明统一放在这里：

- `使用说明.md`：普通用户怎样完成转换。
- `格式说明.md`：支持格式与转换结果说明。
- `构建说明.md`：从源码生成 Windows 程序。
- `发布说明.md`：正式 ZIP 的内部目录。
- `兼容报告.md`：当前验证范围与结论。
- `路线图.md`：多平台后续方向。
- `更新日志.md`：版本变化记录。
- `第三方组件.md`：依赖和许可证信息。
- `研究/`：上游播放器来源与本地分析记录。

### `packaging/windows/`

- `yucedu-converter.spec`：PyInstaller 构建规则。
- `launcher.py`：Windows 打包启动入口。
- `version_info.txt`：可执行文件版本信息。

### `scripts/`

- `build_windows.ps1`：生成 Windows 可执行目录。
- `package_release.ps1`：把构建结果整理成正式 ZIP 并生成 SHA256。
- `check_privacy.py`：检查公开文件中是否出现隐私路径或敏感资料。
- `smoke_test_gui.py`：执行最小界面启动检查。
- `verify_taskbar_icon.py`：检查 Windows 任务栏图标行为。

### `tests/`

保存基于 Python `unittest` 的自动测试。`tests/fixtures/README.md` 只说明夹具要求，真实媒体由本机环境变量选择，不进入仓库。

### `local/`

这是本机专用目录，克隆仓库后可以自行创建：

```text
local/
├─ fixtures/
│  └─ regression.yucedu
└─ players/
   ├─ windows/
   └─ macos/
```

整个目录已被 `.gitignore` 排除，只用于本机回归和兼容研究。

## 五、程序数据流程

```text
用户选择模式、输出目录和文件
            ↓
GUI 创建任务并放入后台工作线程
            ↓
converter 检查输入格式、资源和输出路径
            ↓
按数据块执行转换，持续发送进度
            ↓
先写入临时文件，完成后再替换为正式输出
            ↓
GUI 显示结果，可调用 player 打开文件或所在目录
```

这样设计的主要作用是：避免大文件一次性占满内存；转换未完成时不留下看似完整的输出；后台处理时界面保持响应。

## 六、文件与路径规范

1. 根目录只放项目入口和工具要求的配置，不新增普通说明、截图、样本或临时脚本。
2. 用户与开发说明放入 `docs/`；GitHub 能识别的协作文件放入 `.github/`。
3. Python 源码使用标准 `src` 布局；新模块放入 `src/yucedu_converter/`。
4. 程序运行资源放入 `src/yucedu_converter/resources/`，不要散落到根目录。
5. Windows 打包配置放入 `packaging/windows/`，可执行维护脚本放入 `scripts/`。
6. 文档路径最多保留两层；研究资料统一放入 `docs/研究/`。
7. 文档、源码和配置只使用相对路径、环境变量或示例占位符，不记录个人绝对路径。
8. 本机样本和参考播放器放入 `local/`；真实媒体、账号资料和原播放器不提交 Git。
9. `build/`、`dist/`、`release/`、缓存、日志和转换结果都是生成内容，由 `.gitignore` 排除。
10. 文件版本通过 Git 标签与 `docs/更新日志.md` 管理，不复制出“最终版”“最新版”等平行文件。

## 七、命名规范

- 用户直接看到的说明文档和交付文件优先使用清晰中文名。
- Python 包、模块、函数、变量、配置键和工作流按工具规范使用英文。
- Python 文件使用小写英文与下划线，例如 `check_privacy.py`。
- Python 类使用大驼峰，例如 `ConversionResult`。
- GitHub 仓库、Python 包和发布资源继续使用稳定名称 `yucedu-converter`，避免安装与链接失效。

## 八、生成目录与发布目录

以下目录由脚本生成，可以删除后重新构建，不应提交 Git：

```text
build/                              # PyInstaller 中间文件
dist/YUCEdu双向转换器/              # 可运行程序目录
release/
├─ yucedu-converter-v2.0.1-windows-x64/
├─ yucedu-converter-v2.0.1-windows-x64.zip
└─ yucedu-converter-v2.0.1-windows-x64.zip.sha256.txt
```

发布包内部结构见 [发布说明](发布说明.md)，开发与提交要求见 [参与贡献](../.github/CONTRIBUTING.md)。
